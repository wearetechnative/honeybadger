## Context

The current `check-output` command in RUNME.sh generates a plain text report (`honeybadger-{user}-{date}-report.txt`) by calling `check_os_status()` and `generate_asset_inventory()` from `lib/_library`, then capturing all terminal output via `exec > >(tee "$report_file")`. This works for technical OS/kernel analysis but doesn't provide structured ISO27001 compliance reporting.

The Honeybadger audit system already collects all necessary data:
- `lynis-report.json`: Contains hardening score, firewall status, malware scanner status, and detailed warnings/suggestions
- `blockdevices.txt`: Contains disk encryption information (LUKS detection)
- `screenlock-info.txt`: Contains screen lock configuration and AUTO-LOCK-STATUS markers
- `os-kernel-status.txt` and `asset-inventory.txt`: Generated by existing library functions
- `hardware-serial.txt`: Hardware serial number (collected via dmidecode during audit)
- `os-update-history.txt`: OS update status and last update date (collected via check-os-updates.sh during audit)

The challenge is transforming this technical data into two business-focused reports: a compliance checklist and a prioritized action list.

**Constraints:**
- Must maintain compatibility with existing `check-output` command interface
- Cannot modify the data collection phase (audit command)
- Must work with both directory and tarball inputs (existing behavior)
- Bash-based implementation to match existing codebase

## Goals / Non-Goals

**Goals:**
- Generate ISO27001-compliant markdown reports replacing the current text report
- Provide clear pass/fail compliance status for required controls (including OS update status and hardware asset tracking)
- Classify Lynis findings by severity using market-standard security practices
- Integrate existing OS/kernel and asset inventory data into compliance report
- Include hardware serial numbers for proper asset identification (ISO27001 A.8.1)
- Add OS update status compliance check with 14/30-day thresholds (ISO27001 Policy 8.1)
- Maintain the existing `check-output` command interface and tarball extraction logic

**Non-Goals:**
- Modifying the audit data collection phase (audit command) - serial numbers and OS updates are already collected
- Changing the Lynis scanning process
- Adding new data collection beyond what's already implemented (hardware-serial.txt and os-update-history.txt already exist)
- Supporting other compliance frameworks (NIST, SOC2, etc.) in this change
- PDF generation (HTML/PDF generation already exists for warnings report)
- Interactive reporting or web-based dashboards

## Decisions

### Decision 1: Two separate markdown files instead of one combined report

**Chosen:** Generate `compliance.md` (executive summary) and `actions.md` (technical details) as separate files

**Rationale:**
- Different audiences: compliance.md for management/auditors, actions.md for engineers
- Separation of concerns: compliance status vs remediation tasks
- File size: actions.md could be very large with many Lynis findings
- Allows engineers to focus on actions without reading compliance narrative

**Alternatives considered:**
- Single combined report: Would be too long, mixing executive summary with technical details
- Three files (compliance, actions, system-info): Over-engineering, system info belongs in compliance context

### Decision 2: Replace report.txt entirely instead of adding new files

**Chosen:** Remove `honeybadger-{user}-{date}-report.txt` generation and replace with `compliance.md`

**Rationale:**
- report.txt is ephemeral output, not a stored artifact (backward compatibility not required)
- All useful information from report.txt (OS/kernel status, asset inventory) will be included in compliance.md
- Reduces redundancy and user confusion about which report to review
- Cleaner output directory with purpose-specific files

**Alternatives considered:**
- Keep all three files: Creates redundancy, users won't know which to read
- Redirect report.txt to compliance.md via symlink: Confusing, breaks expected file type (.txt vs .md)

### Decision 3: Bash functions in lib/_library instead of separate scripts

**Chosen:** Add new functions to existing `lib/_library` file

**Rationale:**
- Consistency with existing codebase architecture (check_os_status, generate_asset_inventory already there)
- Shared helper functions and configuration (cache handling, JSON parsing with jq)
- Single source location for all report generation logic
- Easier testing and maintenance

**Alternatives considered:**
- Separate lib/compliance-report.sh: Unnecessary file proliferation, would still need to source _library for helpers
- Inline in RUNME.sh: Would bloat the main script, harder to test

### Decision 4: Keyword-based severity classification for Lynis findings

**Chosen:** Implement severity classification using keyword pattern matching against finding descriptions

**Severity Mapping:**
```
CRITICAL (fix binnen 1 week):
- Test ID PKGS-7392 (vulnerable packages)
- Warnings with keywords: vulnerable, unpatched, outdated, exposed, no encryption, no firewall, authentication failed

HIGH (fix binnen 1 week):
- All other warnings
- Suggestions with keywords: kernel, ssh, remote, permissions, access control, update, patch

MEDIUM (fix binnen 1 maand):
- Default for suggestions not matching other categories

LOW (best effort):
- Suggestions with keywords: consider, optional, recommended
```

**Rationale:**
- Lynis doesn't provide severity levels in its output
- Market-standard approach aligns with ISO27001, NIST, and common vulnerability management practices
- Keyword matching is simple, deterministic, and maintainable
- Bash-friendly implementation (case-insensitive grep/pattern matching)

**Alternatives considered:**
- Manual severity mapping per test ID: Too many test IDs (hundreds), unmaintainable
- Machine learning classification: Overkill, requires Python/external dependencies
- External severity database: Adds dependency, may become stale
- Using Lynis suggestion priority field: Exists but doesn't map well to business impact levels

### Decision 5: Reuse existing check_os_status and generate_asset_inventory outputs

**Chosen:** Read from `os-kernel-status.txt` and `asset-inventory.txt` files instead of regenerating

**Rationale:**
- These files are already generated during check-output execution
- Avoids code duplication and potential inconsistencies
- Simpler implementation (file reading vs reimplementing logic)
- These functions are mature and tested

**Alternatives considered:**
- Call functions directly and capture output: Would require refactoring to return structured data instead of formatted text
- Parse the text files: Already formatted for inclusion in markdown report

### Decision 6: Hardcoded compliance thresholds in code

**Chosen:** Embed thresholds directly in functions (hardening score ≥75, screen lock timeout ≤15 min)

**Rationale:**
- Thresholds are based on ISO27001 policy requirements (fixed, not configurable)
- Simplifies implementation (no config file parsing)
- Thresholds are unlikely to change frequently
- Documented in ISO27001-LAPTOP-COMPLIANCE.md reference document

**Alternatives considered:**
- Configuration file: Over-engineering for static values defined by compliance policy
- Environment variables: Adds complexity, thresholds should not vary per execution

### Decision 7: Include OS update status as compliance check

**Chosen:** Add OS update status check reading from `os-update-history.txt` with compliance thresholds

**Compliance Thresholds:**
- ✅ COMPLIANT: Last update <14 days ago
- ⚠️ WARNING: Last update 14-30 days ago (UPDATE RECOMMENDED)
- ❌ NON-COMPLIANT: Last update >30 days ago (UPDATE REQUIRED)

**Rationale:**
- OS update recency is critical for ISO27001 Policy 8.1 (patch management)
- Data already collected during audit via check-os-updates.sh
- 14-day threshold aligns with industry best practices for security updates
- 30-day threshold matches common compliance requirements
- Provides clear actionable guidance (update recommended vs required)

**Alternatives considered:**
- Skip OS update check: Would miss critical compliance requirement for patch management
- Different thresholds: 14/30 days chosen based on typical security update cycles and ISO27001 guidance
- Weekly update requirement: Too strict for most organizations, would cause alert fatigue

### Decision 8: Include hardware serial numbers in compliance report

**Chosen:** Display hardware serial number in system information section of compliance report

**Rationale:**
- Serial numbers are essential for asset tracking and ISO27001 asset inventory (A.8.1)
- Data already collected during audit via dmidecode
- Enables proper hardware asset identification for compliance audits
- No additional data collection needed (graceful fallback for VMs already implemented)

**Alternatives considered:**
- Omit serial numbers: Would make asset tracking difficult for compliance auditors
- Separate asset report: Over-engineering, serial number naturally belongs in system info section

### Decision 9: Add fallback firewall detection when Lynis reports no firewall

**Chosen:** Enhance `check_firewall_status()` to perform direct iptables/nftables checks when Lynis reports `firewall_active=0`

**Rationale:**
- Lynis produces false negatives on systems with active firewalls it doesn't recognize (NixOS nixos-fw chain, custom iptables/nftables configs)
- Direct rule checking is more reliable than tool detection (ufw/firewalld)
- Prevents compliant systems from being incorrectly flagged as non-compliant (critical for ISO27001 audits)
- NixOS systems commonly affected: nixos-fw chain not recognized by Lynis
- Fallback only runs when Lynis says "no firewall" (minimal performance impact)

**Implementation approach:**
1. First check Lynis `firewall_active` field (primary detection)
2. If Lynis reports 0 (no firewall), run fallback checks:
   - Check for NixOS nixos-fw chain with active rules (`iptables -L nixos-fw -n`)
   - Check for generic iptables rules (non-empty chains beyond default policy)
   - Check for nftables rules (`nft list ruleset`)
3. Return "Yes" if fallback detects active rules, otherwise "No"
4. Document detection method in compliance report (Lynis vs fallback)

**Alternatives considered:**
- Only use direct rule checking: Would miss firewall software that Lynis detects correctly (ufw status, firewalld service)
- Always trust Lynis: Continues to produce false negatives on NixOS and custom configs
- Add NixOS-specific check only: Doesn't solve problem for other custom firewall configs

### Decision 10: Remove HTML/PDF warnings report generation

**Chosen:** Remove `generate_warnings_report()` function and its call from audit command

**Rationale:**
- Redundant: New markdown actions report (`honeybadger-{user}-{date}-actions.md`) provides same information with better filtering (CRITICAL/HIGH only)
- Markdown is more maintainable and version-control friendly than HTML
- HTML/PDF generation adds complexity (wkhtmltopdf dependency for PDF)
- Markdown can easily be converted to HTML/PDF externally if needed (pandoc, etc.)
- Reduces audit output clutter (one actions.md instead of .html + .pdf files)
- Consistent format: All reports now in markdown (compliance.md, actions.md)

**Implementation:**
1. Remove `generate_warnings_report()` function from lib/_library (lines 942-1251)
2. Remove function call from RUNME.sh audit command (line 478-480)
3. Remove console output message "Generating warnings and suggestions report..."

**Alternatives considered:**
- Keep both HTML and markdown: Redundant, adds maintenance burden
- Only remove PDF generation: HTML alone still adds clutter without value
- Convert markdown to HTML automatically: Over-engineering, users can do this externally if needed

## Risks / Trade-offs

**Risk:** Keyword-based severity classification may misclassify some findings
→ **Mitigation:** Keywords chosen based on common security patterns; severity mapping is documented for manual override; users can review actions.md and reprioritize if needed

**Risk:** Large Lynis reports could generate very long actions.md files
→ **Mitigation:** Markdown format with clear section headers allows easy navigation; future enhancement could add filtering/pagination if needed

**Risk:** Changes to Lynis JSON output format could break parsing
→ **Mitigation:** Lynis JSON format is stable; jq queries use safe navigation (`// empty`); existing codebase already depends on this format

**Trade-off:** Removing report.txt means losing the full terminal output capture
→ **Justification:** All meaningful information (OS status, asset inventory, compliance checks) is included in compliance.md; terminal debugging output not needed for compliance reporting

**Trade-off:** Bash keyword matching is less sophisticated than NLP-based classification
→ **Justification:** Simpler implementation, no external dependencies, deterministic behavior, sufficient accuracy for intended use case

**Risk:** Users may expect PDF output for compliance reports
→ **Mitigation:** Markdown is easily converted to PDF via pandoc/external tools; focus on content quality first; PDF generation can be added later if needed

**Risk:** Firewall fallback detection requires sudo/root permissions to run iptables/nftables commands
→ **Mitigation:** check-output runs on audit output files only (doesn't need live system access); fallback detection should read from captured audit data if available; document that live firewall detection requires sudo

**Trade-off:** Fallback firewall detection adds complexity to check_firewall_status()
→ **Justification:** Eliminates critical false negatives that affect compliance reporting accuracy; complexity is isolated to one function; well-tested on multiple firewall configurations

## Migration Plan

**Deployment Steps:**
1. Add new functions to `lib/_library`:
   - `generate_compliance_report()`
   - `generate_lynis_actions_report()`
   - `classify_lynis_severity()`
   - Individual compliance check functions

2. Modify `RUNME.sh check-output()`:
   - Remove `exec > >(tee "$report_file")` terminal capture
   - Replace with calls to new report generation functions
   - Keep existing tarball extraction and fetch_os_releases logic

3. Update documentation (if any exists for check-output)

4. Test with existing audit output directories

**Rollback Strategy:**
- If issues occur, revert RUNME.sh and lib/_library changes
- Old report.txt generation can be restored from git history
- No data loss risk (input files unchanged, only output format changes)

**Testing Approach:**
- Test with real audit output directories from different OS types (NixOS, Ubuntu, Debian)
- Verify all compliance checks work with edge cases (no encryption, no screen lock, etc.)
- Test severity classification with sample Lynis findings
- Verify tarball extraction still works
